Change Request (CR)

CR Title

Update KYC Renewal Date Logic Based on New CDD Renewal Policy (1255) with COI Exclusion List

Change Type
Enhancement

Application
CRMS (Customer Relationship Management System)

Module
KYC / Customer Due Diligence (CDD)

⸻

Description

This change is to update the KYC renewal date calculation based on the new CDD Renewal Policy (1255).

A new configurable database table (HRTC_COUNTRY_LIST) will be created to maintain the list of Countries of Incorporation (COI) that should not follow the new renewal policy.

During KYC calculation, the application will first check whether the customer’s COI exists in the HRTC_COUNTRY_LIST.

* If the country exists in the exclusion list, the existing KYC renewal logic will be applied.
* If the country is not in the exclusion list, the new 1255 renewal logic will be applied.

The change also includes adding the missing countries to the CRMS Country Master and updating the KYC onboarding and renewal date calculation.

⸻

Justification

This change is required to implement the revised CDD Renewal Policy (1255) for KYC renewal date calculation.

The current logic does not support maintaining excluded countries through configuration. Every new country addition or removal requires code changes.

By introducing a configurable exclusion table, future maintenance can be done directly through the database without modifying the application code. This reduces maintenance effort, improves flexibility, and ensures compliance with the latest business requirements.

⸻

Implementation Plan

1. Take backup of the current application and database objects.
2. Deploy the updated SAS program.
3. Create the HRTC_COUNTRY_LIST table.
4. Insert the configured exclusion countries into the table.
5. Update the CRMS Country Master with the missing countries.
6. Deploy the database scripts.
7. Restart application services if required.
8. Verify deployment logs.
9. Perform smoke testing.
10. Validate KYC renewal calculation in UAT.
11. Deploy the changes to Production after UAT approval.

⸻

Business Impact

If the change is not implemented:

* KYC renewal dates will not follow the new CDD Renewal Policy (1255).
* Customers may receive incorrect KYC expiry dates.
* Manual corrections may be required by business users.
* Future country updates will require application code changes instead of simple database updates.

There is no expected business downtime during implementation.

⸻

Technical Impact

If the change is not implemented:

* The application will continue using the existing KYC renewal logic.
* New CDD renewal rules will not be applied correctly.
* Country exclusions cannot be managed through configuration.
* Every country update will require a new code deployment.

This change includes updating the SAS program, creating a new database table, and updating the Country Master. No infrastructure changes are involved.

⸻

KYC Alert Generation Logic

The attached code shows that the system generates KYC Expiry Alerts based on the following conditions:

1. Expiry Date Check

The system checks the KYC Expiry Date (KYC_EXPIRY_DATE) and generates an alert only if it falls between the calculated date range.

PGORCL.KYC_EXPIRY_DATE BETWEEN (V_LST_RUN) AND (V_CUR_RUN)

The date range is calculated using:

V_CUR_RUN := V_CUR_RUN + DAYS_VALUE;
V_LST_RUN := V_LST_RUN + DAYS_VALUE;

The procedure is executed for:

* 0 days
* 30 days
* 60 days
* 90 days

So, alerts are generated for KYC records expiring today, in 30 days, 60 days, and 90 days.

⸻

2. Zero Notification Flag

The alert is generated only when:

ZERO_NOTIFICATION_FLG = 'NO'
OR
ZERO_NOTIFICATION_FLG IS NULL

This means customers who have disabled zero notifications are excluded.

⸻

3. Relationship Status

Only active customer relationships are considered.

NVL(X_RELATIONSHIP_STATUS,
GB_PCK.V_REL_STATUS_ACTIVE)
=
GB_PCK.V_REL_STATUS_ACTIVE

⸻

4. Case Status

Only cases with status ‘C’ are processed.

CASE_STATUS_CD = 'C'

⸻

5. Alert Creation

If all the above conditions are satisfied, the system creates the alert using:

GENERATE_XXX_ALERTS

⸻

6. Alert Frequencies

The procedure is called four times:

GENERATE_KYCEXPXX_ALERTS(...,0);
GENERATE_KYCEXPXX_ALERTS(...,30);
GENERATE_KYCEXPXX_ALERTS(...,60);
GENERATE_KYCEXPXX_ALERTS(...,90);

Therefore, KYC alerts are generated for:

* 0 days – Expiring today
* 30 days – Expiring in 30 days
* 60 days – Expiring in 60 days
* 90 days – Expiring in 90 days

⸻

* The customer’s KYC Expiry Date falls within the configured alert window (0, 30, 60, or 90 days).

    * DAYS_VALUE is passed as 0, 30, 60, or 90.
    * This means the system checks for IDs expiring today, in 30 days, 60 days, and 90 days.

CASE_TYPE_CD = 'CDDINV'
AND TO_DATE(X_EXPIRY_DATE,'MM/DD/RRRR')
BETWEEN V_LST_RUN AND V_CUR_RUN

    OR
    * Case Type = ‘AMLINV’ and the Customer ID Expiry Date (X_CR_EXPIRY_DATE) falls between the calculated date range.

CASE_TYPE_CD = 'AMLINV'
AND TO_DATE(X_CR_EXPIRY_DATE,'MM/DD/RRRR')
BETWEEN V_LST_RUN AND V_CUR_RUN

    * Customer relationship must be Active.

NVL(X_RELATIONSHIP_STATUS, GB_PCK.V_REL_STATUS_ACTIVE)
